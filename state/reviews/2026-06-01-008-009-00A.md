## #1 — (HIGH) Failed / over-cap Gemini attempts never record token spend → $2 cost guard undercounts

**File:** `platform/src/lib/gemini/client.ts:184` (record), with leaks at `:175` and `:200`

`costAccumulator.record(...)` runs **only** on the success path, inside the `try` after
`opts.schema.parse(...)` succeeds. Two paths spend real tokens but record nothing:

- `assertWithinOutputCap` throws at `client.ts:175` → that call's tokens are dropped.
- All repair attempts fail → `throw new SchemaValidationError(lastIssue)` at `client.ts:200`
  → tokens from **every** attempt (initial + repairs) are dropped.

Failure/repair loops are exactly when per-site spend balloons (each repair is a full extra
dispatch), yet they're invisible to the accumulator. A later `assertCanSpend` / `wouldExceed`
check then sees a total lower than what was actually billed, so a site can exceed the $2 cap
undetected.

**Fix:** accumulate usage regardless of parse/cap outcome. Record into the accumulator in a
path that runs even when the attempt fails (e.g. record per-attempt usage as soon as
`readUsage` returns, or in a `finally`), instead of only on the successful return. Decide
whether `record()` should split into "record usage" (always) vs "record image count" (only on
a successful image) so the image counter doesn't over-count failed image calls.

**Note:** `record()` currently both tallies tokens AND bumps `imageCount`. When refactoring,
keep image-count increments on success only (a failed image call shouldn't consume image quota),
but token spend should always count.

## #2 — (MEDIUM) Cost guard is recorded but never *enforced* in the client path

**File:** `platform/src/lib/gemini/cost.ts:85` (`assertCanSpend`), `:95` (`assertCanGenerateImage`)

Both methods are defined but have **no callers** (grep confirms). `generateJSON` records cost
after the fact and never consults the cap or the 3-image counter before dispatching. The client
is documented as "the one place every AI call goes through… enforces token budgets," but the
cost/image caps are left entirely to the (still-stubbed) pipeline caller.

Combined with #1, nothing in the wired path can currently halt before $2 or past 3 images.

**Fix / decision needed:** either
- (a) have `generateJSON` call `assertCanSpend(preEstimate)` before dispatch and
  `assertCanGenerateImage()` before an image call, so the guard can't be forgotten downstream; or
- (b) if enforcement is deliberately the caller's job, document that contract explicitly and
  ensure the pipeline tickets (010–021, esp. 020 generation / 022 images) wire it in. Add a test
  that proves a site is halted at the cap.

The token caps (`assertWithinInputCap` / `assertWithinOutputCap`) ARE wired and enforced — this
finding is specifically about the **dollar** cap and the **image-count** cap.

## #3 — (MEDIUM) Deletion re-read uses `.single()` and swallows its error

**File:** `platform/src/lib/account/service.ts:152`

The already-pending branch of `requestAccountDeletion` re-reads via `.single()` and ignores the
returned `{ error }`. Every other read in this file uses `.maybeSingle()` and checks `error`
(see `getAccountSettings` at `:43-52`). On a transient DB error here, `.data?.deletion_requested_at`
is `undefined` → `requestedAt` becomes `null` → the function throws the misleading
`"Failed to schedule account deletion"` (`:157`) instead of surfacing the real error.

**Fix:** use `.maybeSingle()`, capture `{ data, error }`, and `throw error` on failure —
consistent with the rest of the module. Then derive `requestedAt` from `data`.
