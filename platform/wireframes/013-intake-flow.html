Ordering decision — read first

PRD §4.1 numbers template pick as step 9 (before Round-1). We reorder template to the end of this flow, on purpose: the template preview is only compelling once we have the confirmed brand color and the uploaded/generated logo — both of which are settled in steps 2–3 here. Showing an empty grey preview first undersells "templates are the product" (§7.4). The five screens run:

1. Quick questions (asked while the 012 scrape runs — §4.1.8) → 2. Round-1 confirm-or-correct (§4.1.10, §8.3) → 3. Asset upload + logo (§4.1.11, §6.8) → 4. Round-2 questions (§4.1.12, §8.3) → 5. Template selection (§4.1.9, §6.1).

Every screen auto-saves to intake_data / assets / sites before advancing, so refresh and "save & continue later" resume exactly here — same resume model 010 already established.

Why confirm-or-correct beats blank fields

§8.3 calls this a "massive completion-rate difference." Blank forms make the advisor do extraction work we already did; a pre-filled "✓ Yes / Edit" turns a 12-field form into mostly one tap. Conservative buyers (§7.1) abandon long forms — this is the single biggest completion lever in the flow, so we don't regress it.

Confidence surfaced, not hidden

Each field shows the confidence + sources already stored on IntakeField (012). Low-confidence (amber) and missing (red) rows draw the eye to exactly what needs a human, so the advisor corrects the 2–3 that matter instead of re-reading all 12.

Row states
  • Confirmed (green ≥ .85) — collapsed, one Edit button
  • Low-confidence (amber .4–.85) — nudged to check
  • Missing (red, conf 0) — "Add", not "Edit"
  • Editing — inline input, Save/Cancel, writes on save

Preview approach

Each card is a real (mini) render of the template's hero/type system, tinted with the extracted brand color (012) so the advisor sees their site, not a generic mock. No color found → documented default (emerald). "Templates are the product" (§7.4) — these previews must look polished, not grey boxes (full-fidelity in build).

Persistence & the one-site rule

Selection writes sites.template_id (trust|modern|boutique). Exactly one site per account in v1 — we upsert, never insert a second. Locked after build (§3.2); the 029 edit-chat guardrails enforce no-switching later.

States
  • Three cards, one selected (radio-exclusive)
  • Brand-color preview vs. default-color preview
  • Continue disabled until one is chosen
  • Post-build: read-only (out of scope here)
